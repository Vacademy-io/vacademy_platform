# Developer Database Access

How engineers get read-only (and, when justified, read-write) access to the
production Postgres cluster on Hetzner. Use this for query debugging, analytics,
and incident triage — **not** for application traffic, which goes through
`pgbouncer` from inside the k3s cluster.

For the underlying topology (boxes, IPs, replication), see
[`hetzner/TOPOLOGY.md`](./hetzner/TOPOLOGY.md). Quick summary:

| Role | Private IP | Public IP (SSH only) | Purpose |
|---|---|---|---|
| `db-primary` | `10.0.0.4` | `5.223.55.54` | Accepts writes. Use only for RW work. |
| `db-standby` | `10.0.0.5` | `5.223.53.24` | Streaming replica. **Default target for read-only work.** |

Postgres listens on `:5432` but the `fw-db` Hetzner Cloud Firewall only allows
`5432` from inside the `10.0.0.0/16` private network. **There is no direct
public Postgres access.** Developers reach the DB by SSH-tunneling through the
DB host's public IP.

---

## 1. Roles & access model

Two group roles exist on both DB hosts (replicated automatically by streaming
replication of `pg_authid`):

### `vacademy_readonly`

- `CONNECT` on every service DB (6 of them)
- `USAGE` on `schema public`
- `SELECT` on every existing table and sequence
- `EXECUTE` on every function
- `ALTER DEFAULT PRIVILEGES` ensures **future** tables created by `vacademy` or
  `postgres` also get `SELECT` automatically.
- **Denied:** `INSERT`, `UPDATE`, `DELETE`, `TRUNCATE`, any DDL.

This is the default for engineers, analysts, support, and AI/data work.

### `vacademy_readwrite`

- Everything in `vacademy_readonly`, plus:
- `INSERT`, `UPDATE`, `DELETE` on existing tables
- `USAGE` + `SELECT` + `UPDATE` on sequences
- **Still denied:** `TRUNCATE`, `CREATE`, `DROP`, `ALTER TABLE`, schema changes,
  superuser ops.

This role exists for narrow, time-boxed fixes (data repair, hotfix backfill).
**Default answer is no.** Granted by the on-call DBA / platform lead only.

### Group → user mapping

Individual developer accounts (`dev_<name>`) are created with `LOGIN`,
`NOSUPERUSER`, and group membership in **exactly one** of
`vacademy_readonly` or `vacademy_readwrite`. The group roles themselves are
`NOLOGIN` and are never logged into directly.

### Security boundaries

- No public Postgres port. Firewall rule `fw-db` allows `5432` from
  `10.0.0.0/16` only.
- Dev users have no replication, no superuser, no `BYPASSRLS`.
- Application traffic uses the `vacademy` role via `pgbouncer` (separate code
  path); revoking a dev account never affects the app.
- Passwords are 24 random url-safe-base64 chars, generated by the management
  script. Rotate every 90 days (or sooner if a laptop is lost).
- Credential files on `db-primary` live in `/root/vacademy-migration/dev-users/`
  with mode `600`. They are **not** in git and must never be committed.

## 2. How to request access

1. Open a ticket / Slack the platform team with:
   - Your laptop SSH public key (must already work against `db-primary` and
     `db-standby` SSH; usually it already does if you can reach the cluster).
   - Justification (read-only is approved by default; read-write needs an
     explicit reason and an expiry).
2. Platform admin runs, on `db-primary`:

   ```
   ssh root@5.223.55.54
   /root/vacademy-migration/manage-dev-db-user.sh add dev_<yourname> ro
   ```

   For read-write (rare):

   ```
   /root/vacademy-migration/manage-dev-db-user.sh add dev_<yourname> rw
   ```

3. Admin shares the username + password via the email block printed by the
   script (use 1Password / a secure channel, not Slack DM).
4. Admin lists current dev users any time with:

   ```
   /root/vacademy-migration/manage-dev-db-user.sh list
   ```

5. Removal:

   ```
   /root/vacademy-migration/manage-dev-db-user.sh remove dev_<yourname>
   ```

6. Rotation (same script, new password, same group):

   ```
   /root/vacademy-migration/manage-dev-db-user.sh rotate dev_<yourname>
   ```

Streaming replication propagates the role and grants to `db-standby` within a
few seconds — no second action needed.

## 3. Connection guide (SSH tunnel)

You connect by tunneling Postgres over your existing SSH connection to the
relevant DB host. You do **not** need a firewall change.

### Read-only — tunnel to db-standby (preferred)

```
ssh -fNL 15432:10.0.0.5:5432 root@5.223.53.24
```

- `-f` backgrounds SSH after auth.
- `-N` means "no remote command, tunnel only".
- `-L 15432:10.0.0.5:5432` forwards your laptop's `localhost:15432` to
  `10.0.0.5:5432` (db-standby's private IP) via the SSH server.

We use port **15432** locally to avoid clashing with any local Postgres you
might be running.

### Read-write — tunnel to db-primary

```
ssh -fNL 15432:10.0.0.4:5432 root@5.223.55.54
```

Same shape, just different SSH host and target. **Only open this tunnel when
you actually need to write.** When you're done, kill it:

```
pkill -f 'ssh.*-fNL.*15432:10.0.0.4:5432'
```

### Then connect

```
PGPASSWORD='...' psql -h localhost -p 15432 -U dev_<yourname> -d auth_service
```

Or paste the connection string into your GUI tool (see section 6).

## 4. Connection string examples

Six service databases live on the cluster. The standard form is:

```
postgresql://dev_<yourname>:<password>@localhost:15432/<db_name>
```

| Service | DB name |
|---|---|
| Auth | `auth_service` |
| Admin core | `admin_core_service` |
| Assessment | `assessment_service` |
| Community | `community_service` |
| Media | `media_service` |
| Notification | `notification_service` |

Examples (replace `<password>`):

```
postgresql://dev_alice:<password>@localhost:15432/auth_service
postgresql://dev_alice:<password>@localhost:15432/admin_core_service
postgresql://dev_alice:<password>@localhost:15432/assessment_service
postgresql://dev_alice:<password>@localhost:15432/community_service
postgresql://dev_alice:<password>@localhost:15432/media_service
postgresql://dev_alice:<password>@localhost:15432/notification_service
```

> The `TOPOLOGY.md` doc lists 6 service DBs. Earlier internal write-ups
> sometimes said "5"; the live cluster (and `manage-dev-db-user.sh`) carry the
> full 6. `community_service` is the one occasionally missed in old docs.

## 5. Best practices

- **Never write to prod from a developer account unless an incident
  on-caller has approved it in writing.** RW exists for emergencies, not for
  convenience. Prefer migrations through the service.
- **Send analytics / heavy reads to the standby (`db-standby`, `10.0.0.5`)**,
  not the primary. The standby is sized smaller (CCX13 → CPX22) so don't run
  pathological scans there either; use `EXPLAIN` first.
- **Use transactions when poking around.** Wrap exploratory work in
  `BEGIN; ... ROLLBACK;` even on read-only accounts — it disciplines you, and
  the day you switch to a RW account you won't have to retrain the habit.
- **Rotate passwords every 90 days** (or immediately if a laptop is lost,
  someone leaves the team, or a creds file appears in chat). Use:

  ```
  /root/vacademy-migration/manage-dev-db-user.sh rotate dev_<yourname>
  ```

- **Close tunnels when done.** A backgrounded SSH tunnel is an open path into
  prod. `pkill -f 'ssh.*-fNL.*15432'` kills all your tunnels at once.
- **Never commit creds.** Not in `.env` files, not in scripts, not in test
  fixtures. The script saves them with mode `600` on `db-primary` for the
  admin's reference only.
- **One user per human.** No shared accounts. If three people need access,
  create three users — that's what the script is for.
- **Standby lag matters for write-then-read patterns.** If you just wrote on
  the primary and you `SELECT` on the standby, you might not see your row for
  a few hundred ms. Read your own writes on the primary.

## 6. Common tools setup

For every GUI tool below, the SSH-tunnel + `localhost:15432` flow is the same.
Most tools can either use a pre-opened tunnel or open one themselves via an
"SSH tunnel" option in the connection settings.

### psql

Bundled with Postgres (`brew install libpq` on macOS, then `brew link --force
libpq`). With an external tunnel already open:

```
PGPASSWORD='<password>' psql -h localhost -p 15432 -U dev_<yourname> -d auth_service
```

To skip `PGPASSWORD`, put the line in `~/.pgpass` (mode `600`):

```
localhost:15432:*:dev_<yourname>:<password>
```

### TablePlus

`Create a new connection` → `PostgreSQL`.

- Host: `localhost`
- Port: `15432`
- User: `dev_<yourname>`
- Password: `<password>`
- Database: `auth_service` (one connection per DB, duplicate for others)
- SSH Tunnel: leave OFF (you opened it externally), or use TablePlus's "Over
  SSH" tab with host `5.223.53.24`, user `root`, key file pointed at your
  laptop key — then the inner Postgres host becomes `10.0.0.5` and port
  `5432`.

### pgAdmin

`Register → Server`.

- General → Name: `Vacademy prod (standby)`
- Connection → Host: `localhost`, Port: `15432`, User: `dev_<yourname>`
- SSH Tunnel tab: either off (you opened one), or
  - Tunnel host: `5.223.53.24`
  - Username: `root`
  - Authentication: identity file (your key)
  - Then Connection → Host: `10.0.0.5`, Port `5432`

### DBeaver

`New Database Connection → PostgreSQL`.

- Main tab: Host `localhost`, Port `15432`, Database `auth_service`,
  Username `dev_<yourname>`, Password `<password>`.
- SSH tab (alternative to external tunnel): enable "Use SSH Tunnel", Host
  `5.223.53.24`, User `root`, Authentication "Public Key", point at your key.
  Then on Main tab put `10.0.0.5` / `5432`.

### DataGrip / IntelliJ

`+ → Data Source → PostgreSQL`.

- General tab: Host `localhost`, Port `15432`, Database `auth_service`,
  User `dev_<yourname>`, Password `<password>`.
- SSH/SSL tab: alternatively, enable "Use SSH tunnel", Proxy host
  `5.223.53.24`, port `22`, user `root`, key pair, then main host
  `10.0.0.5` / `5432`.

### Beekeeper, Postico, ...

Any Postgres GUI works. The pattern is identical: either point at
`localhost:15432` with an external tunnel, or use the tool's built-in SSH
tunnel against `root@5.223.53.24` and target `10.0.0.5:5432`.

## 7. Quick reference (psql one-liners)

```
\l                    -- list databases
\dt                   -- list tables in current DB (public schema)
\dt+ schema.*         -- list tables in a specific schema, with size
\d table_name         -- describe one table (columns, indexes, FKs, triggers)
\dn                   -- list schemas
\du                   -- list roles (you'll see vacademy_readonly etc.)
\dx                   -- installed extensions (expect pgvector, pgcrypto)
\timing on            -- show query timing
\x                    -- toggle expanded display (great for wide rows)
\q                    -- quit

-- Common counts
SELECT count(*) FROM <table>;

-- Find tables by name fragment
SELECT schemaname, tablename FROM pg_tables
 WHERE tablename ILIKE '%user%' AND schemaname = 'public';

-- Top 10 biggest tables in the DB
SELECT relname, pg_size_pretty(pg_total_relation_size(c.oid)) AS size
  FROM pg_class c
  JOIN pg_namespace n ON n.oid = c.relnamespace
 WHERE n.nspname = 'public' AND c.relkind = 'r'
 ORDER BY pg_total_relation_size(c.oid) DESC
 LIMIT 10;

-- Long-running queries (run on primary)
SELECT pid, now() - query_start AS age, state, query
  FROM pg_stat_activity
 WHERE state <> 'idle'
 ORDER BY age DESC NULLS LAST;

-- Confirm you are on the standby
SELECT pg_is_in_recovery();   -- t on standby, f on primary

-- Replication lag (run on standby)
SELECT now() - pg_last_xact_replay_timestamp() AS replica_lag;
```

## 8. Troubleshooting

### `Connection refused` on `localhost:15432`

The SSH tunnel is not open. Run:

```
ssh -fNL 15432:10.0.0.5:5432 root@5.223.53.24
```

then `lsof -nP -iTCP:15432 | grep LISTEN` should show `ssh`.

### Tunnel exits immediately

You don't have SSH access to the DB host. Confirm with
`ssh root@5.223.53.24 'hostname'`. If that fails, you need your SSH key added
to `~/.ssh/authorized_keys` on the host — open a ticket.

### `FATAL: password authentication failed for user "dev_..."`

The password is wrong, or you're hitting the wrong DB. Re-check the password
the admin sent (no trailing newline), and confirm you're connecting via
`localhost:15432`, not directly to a remote host. If still failing, ask the
admin to `rotate` and re-send.

### `ERROR: permission denied for table <X>` on SELECT

You're connected with a dev account that lost group membership somehow. Run:

```
SELECT * FROM pg_auth_members
 WHERE member = (SELECT oid FROM pg_roles WHERE rolname = 'dev_<yourname>');
```

You should see one row with `vacademy_readonly` (or `_readwrite`). If not,
ask the admin to re-`add` you (it's idempotent and re-asserts membership).

### `ERROR: cannot execute INSERT in a read-only transaction`

You connected to **db-standby** (the replica) and tried to write. This is
expected: even with a read-write role, the standby is physically read-only.
Switch your tunnel to db-primary:

```
pkill -f 'ssh.*-fNL.*15432'
ssh -fNL 15432:10.0.0.4:5432 root@5.223.55.54
```

### `ERROR: permission denied for table <X>` on INSERT/UPDATE/DELETE

You're a `vacademy_readonly` user trying to write, even on the primary. This
is the intended grant boundary — request `vacademy_readwrite` (with
justification) if you actually need to write.

### `ERROR: permission denied for schema public` on CREATE / ALTER

Neither role grants DDL. Schema changes go through service migrations, not
ad-hoc dev access.

### Stale tunnel after laptop sleep

SSH tunnels die when your laptop sleeps. Re-run the `ssh -fNL ...` command.
Add `ServerAliveInterval 60` to `~/.ssh/config` to keep them up longer.

### Multiple GUIs hanging onto the same tunnel

Each GUI opens its own connection over the single tunnel — that's fine. But
if you close the tunnel and a GUI still has a connection from earlier, the
GUI may show "broken pipe" only on the next query. Close and reopen the GUI's
connection after re-running the tunnel.

### How do I know if I'm on primary or standby?

```
SELECT inet_server_addr(), inet_server_port(), pg_is_in_recovery();
```

`pg_is_in_recovery() = t` means standby. `inet_server_addr()` shows the
private IP (`10.0.0.4` primary, `10.0.0.5` standby).

## 9. Topology reference

For the full picture — boxes, private network, firewalls, k3s cluster, and
backup setup — see [`hetzner/TOPOLOGY.md`](./hetzner/TOPOLOGY.md). The
load-bearing facts for dev access:

- **db-primary**: `5.223.55.54` (public, SSH only) / `10.0.0.4` (private,
  Postgres). Accepts writes.
- **db-standby**: `5.223.53.24` (public, SSH only) / `10.0.0.5` (private,
  Postgres). Streaming hot-standby; physically read-only.
- **Firewall**: Postgres `5432` only reachable from `10.0.0.0/16`. Public
  ingress goes through SSH.
- **Replication**: physical streaming. Roles, grants, and password rotations
  on the primary propagate to the standby automatically.
