# Stage Server — Developer Guide

How to deploy to, inspect, and configure the **standalone stage server**.

Stage is a single Hetzner box running the whole platform on **k3s** (single-node
Kubernetes) via the `vacademy-services` Helm chart — bundled Postgres, Redis, all
backend services, the AI service, and both frontends, behind a Let's Encrypt
ingress. It mirrors how a client "standalone" install runs.

| | |
|---|---|
| **Admin dashboard** | https://admin-stage.vacademy.io |
| **Learner app** | https://app-stage.vacademy.io |
| **API** | same-origin under each host (e.g. `https://admin-stage.vacademy.io/auth-service/...`) |
| **Server** | `root@188.245.114.166` (k3s) |
| **Helm release** | `vac` (namespace `default`) |
| **Chart on box** | `/root/vacademy-services` |
| **Stage config/secrets** | `/root/vacademy-services/values.secret.yaml` (on the box, gitignored) |

## 1. Deploy to stage (pick a branch)

Use the **Actions → "Deploy to Stage Server (standalone k3s)"** workflow
(`.github/workflows/deploy-stage-server.yml`). Click **Run workflow** and set:

| Input | What it does |
|---|---|
| **branch** | Which branch's **chart/config** to deploy (and which branch the **frontends** are built from). Usually `main`. |
| **image_tag** | Which **backend** image tag to run. `latest` = current `main`. For a branch, build that tag first (see below). |
| **optional_services** | Comma-sep list to enable: `assessment,ai,notification,community`. Mandatory ones (auth, admin-core, media) are always on. |
| **rebuild_frontends** | On by default — rebuilds the admin + learner images **from `branch`** before deploying. Uncheck if only the backend changed. |

What the workflow does, in order:
1. **`build-frontends`** — builds both FE images from `branch`, pushes `:image_tag` + `:latest` to ECR. (Skipped if `rebuild_frontends` is off.)
2. **`deploy`** — syncs the chart to the box (your `values.secret.yaml` is preserved), runs `helm upgrade`, then **force-restarts every deployment** so pods re-pull `:latest`.

> **Why the forced restart:** `:latest` is a moving tag. A same-tag `helm upgrade`
> produces an identical pod spec, so Kubernetes does nothing and the new image is
> never pulled. The workflow runs `kubectl rollout restart` to guarantee a re-pull.

### Deploying a feature branch (not main)

Backend images are **not** built by the deploy workflow — build them first:

1. Run the per-service build workflow(s) for what you changed — e.g.
   **"Deploy Auth Service to LKE"** → Run workflow → set `branch` = your branch and
   `image_tag` = a unique tag (e.g. the branch name). On a manual run these only
   **build & push** the image; they do not touch the cloud LKE cluster.
2. Run **"Deploy to Stage Server"** with `branch` = your branch and
   `image_tag` = that same tag. (Frontends build from the branch automatically.)

> Pinning *all* backend services to one `image_tag` means that tag must exist for
> every enabled service. For a partial branch test, the simplest path is to merge
> to `main` and deploy `latest`, or build the few changed services under the same tag.

## 2. Edit an environment variable for stage

The env is generated by the chart, so the **durable** way to change a value is to
edit `values.secret.yaml` on the box and re-apply — not to hand-edit a ConfigMap
(that gets overwritten on the next deploy).

**Which file/key?** Find the env var in the chart to see which value feeds it:
- **Non-secret** (URLs, regions, bucket names, toggles) → `templates/configmap.yaml`, fed by `.Values.env.*`
- **Secret** (keys, passwords, tokens) → `templates/secret-env.yaml`, fed by `.Values.secrets.*`

Common ones:

| Env var | Set in `values.secret.yaml` under |
|---|---|
| `OPENROUTER_API_KEY` | `secrets.api.openrouter` |
| `DEEPSEEK_API_KEY` | `secrets.api.deepseek` |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | `secrets.oauth.googleClientId` / `...googleClientSecret` |
| `AWS_MAIL_USERNAME` / `AWS_MAIL_PASSWORD` (SES) | `secrets.ses.mailUsername` / `...mailPassword` |
| `WHATSAPP_ACCESS_TOKEN_VIDYAYATAN` | `secrets.whatsapp.accessToken` |
| `S3_AWS_ACCESS_KEY` / `S3_AWS_ACCESS_SECRET` | `secrets.s3.accessKey` / `...accessSecret` |
| `AWS_BUCKET_NAME` / `AWS_S3_PUBLIC_BUCKET` | `env.s3.bucket` / `env.s3.publicBucket` |
| `S3_AWS_REGION` / `AWS_REGION` | `env.s3.region` |
| `MAIL_HOST` / `SES_SENDER_EMAIL` | `env.mail.host` / `env.ses.senderEmail` |
| `CLOUD_FRONT_URL` | `env.cloudFrontUrl` |

### Durable edit (recommended)

```bash
ssh root@188.245.114.166
cd /root/vacademy-services
nano values.secret.yaml          # add/edit the key, e.g.:
                                 #   secrets:
                                 #     api:
                                 #       openrouter: "sk-or-..."

export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
helm upgrade --install vac . \
  -f values.yaml -f values-standalone.yaml -f values.secret.yaml --wait

# pick up the new env (Secret/ConfigMap change alone won't restart pods):
kubectl rollout restart deploy --all
kubectl get pods -w
```

Re-running the **"Deploy to Stage Server"** workflow afterward is also fine — it
keeps your edited `values.secret.yaml` and re-applies it.

> A brand-new env var that isn't wired into the chart yet must first be added to
> `templates/configmap.yaml` (non-secret) or `templates/secret-env.yaml` (secret)
> and committed — `values.secret.yaml` can only override keys the templates render.

### Quick / temporary edit (for testing — overwritten on next deploy)

```bash
ssh root@188.245.114.166
export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
kubectl set env deployment/ai-service OPENROUTER_API_KEY=sk-or-...   # restarts that deployment
```
This lasts until the next `helm upgrade` / deploy, which resets it to the chart value.

## 3. Common operations

```bash
ssh root@188.245.114.166
export KUBECONFIG=/etc/rancher/k3s/k3s.yaml   # add to ~/.bashrc to skip this each time

kubectl get pods                      # health of everything
kubectl logs -f deploy/auth-service   # tail a service (auth-service, ai-service, admin-core-service, ...)
kubectl rollout restart deploy/ai-service   # restart one service
kubectl describe pod <pod>            # why a pod is Pending/CrashLooping (events, OOM)
kubectl get certificate               # Let's Encrypt TLS status (READY=True)
helm history vac                      # deploy revisions
helm rollback vac <REVISION>          # roll back a bad deploy
kubectl exec -it postgres-0 -- psql -U postgres   # DB shell (bundled Postgres; it's a StatefulSet)
```

Deployment names (for `logs` / `rollout restart`): `auth-service`,
`admin-core-service`, `media-service`, `assessment-service`, `community-service`,
`notification-service`, `ai-service`, `admin-frontend`, `learner-frontend`,
`redis-master`. The bundled DB is the StatefulSet **`postgres`** (pod `postgres-0`),
so `rollout restart deploy --all` does not touch it.

## 4. Bootstrapping a NEW standalone box

One-time, on a fresh Linux server: `sudo ./install.sh` (installs k3s + Helm +
cert-manager, prompts for domain/keys, writes `values.secret.yaml`, deploys).
Point `admin.<domain>` and `app.<domain>` DNS A-records at the box; TLS issues
itself. After that, day-to-day deploys go through the GitHub Actions workflow.

> ⚠️ **Known mismatch:** `install.sh` creates the Helm release as **`vacademy`**,
> but the CD workflow upgrades **`vac`**. On the existing stage box the release is
> `vac`. Align these before bootstrapping a new box that will also use the CD
> workflow, or the workflow will create a second, conflicting release.
