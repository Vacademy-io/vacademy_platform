# Sunday Cutover Playbook — Linode → Hetzner

**Date:** Sunday (TBD), starting at **T+0** (pick a low-traffic hour; suggest 02:00 IST).
**Window budget:** 90 min plan, 120 min hard cap. Rollback if cap is hit.

## Roles

- **Driver** — runs commands, has SSH to Linode and Hetzner, has Cloudflare API token loaded. (You.)
- **Observer** — watches logs, runs smoke tests, calls rollback. (Me / second person.)

## Pre-flight (T-30 min, before window)

Verify Saturday's dry-run is still healthy. Bail before announcing downtime if any of these fail.

```bash
# Driver workstation — confirm both Helm value overlays exist and are fresh.
# These are generated by hetzner/export-linode-prod-config.sh on Friday from
# the live Linode ConfigMap + Secret; the helm upgrade at T+40 below consumes
# both files. A missing or stale values.secret.yaml here is a HARD STOP —
# without the verbatim jwtSecretKey, every active session and every
# outstanding email/invite token is invalidated the moment Hetzner starts
# serving traffic (CLAUDE.md: "Java services HARDCODE jwtSecretKey ... NEVER
# randomize").
ls -l vacademy_devops/hetzner/values-prod-hetzner.yaml \
      vacademy_devops/hetzner/values.secret.yaml
# Both present; values.secret.yaml mode 0600; mtime within the last ~48h.

# Spot-check that the jwtSecretKey fingerprint still matches Linode:
grep -m1 jwtSecretKey vacademy_devops/hetzner/values.secret.yaml | cut -c1-40
kubectl --context linode-prod -n default get secret vacademy-secrets \
  -o jsonpath='{.data.JWT_SECRET_KEY}' | base64 -d | head -c12; echo
# The 12-char prefixes must match. If they don't, re-run export-linode-prod-config.sh
# BEFORE proceeding.

# Hetzner cluster:
ssh root@<k3s-node-1>
export KUBECONFIG=/etc/rancher/k3s/k3s.yaml
kubectl get nodes                       # all 3 Ready
kubectl get pods -A                     # nothing CrashLoopBackOff
helm list                               # 'vac' release present, deployed (with empty DB)

# Hetzner DB primary:
ssh root@<db-primary>
sudo -u postgres psql -c "\l+"          # all per-service DBs present, schemas restored
sudo -u postgres psql -c "SELECT pg_is_in_recovery();"  # false on primary

# Hetzner DB standby:
ssh root@<db-standby>
sudo -u postgres psql -c "SELECT pg_is_in_recovery();"  # true
sudo -u postgres psql -c "SELECT pg_last_wal_replay_lsn();"  # advancing

# Cloudflare TTL is 60s on every record we'll flip:
dig +short admin.vacademy.io @1.1.1.1
dig +short app.vacademy.io @1.1.1.1
# Both should answer with current Linode LB IP, TTL <120s in the response (`+ttlunits`)
```

**Gate:** all green → proceed at T+0. Any red → reschedule.

### T+0 — Maintenance page up, freeze writes

Put the static maintenance page in front of users **first**, so they see a consistent state
the moment we touch anything.

> **Pre-flight (DO BEFORE T+0):** Verify your kubeconfig has BOTH contexts:
> ```bash
> kubectl config get-contexts
> # Must list both `linode-prod` and `hetzner-prod`. If either is missing,
> # configure them now and abort the cutover until both are present.
> kubectl --context=linode-prod get nodes   # must return Linode node names
> kubectl --context=hetzner-prod get nodes  # must return k3s-node-1/2/3
> ```
> Also confirm the maintenance assets exist on disk before the window starts:
> `ls hetzner/maintenance-page.html hetzner/maintenance-ingress.yaml`
> (HTML splash + Linode-side Ingress + ConfigMap matching admin.vacademy.io /
> app.vacademy.io / backend.vacademy.io, serving 503 + HTML for all paths.)
>
> **Back up current Linode Deployments + Ingress so rollback can restore exact state:**
> ```bash
> mkdir -p /tmp/cutover
> kubectl --context=linode-prod -n default get deploy -o yaml \
>   > /tmp/cutover/linode-pre-cutover-deploys.yaml
> kubectl --context=linode-prod -n default get ingress -o yaml \
>   > /tmp/cutover/linode-pre-cutover-ingress.yaml
> kubectl --context=linode-prod -n default get hpa,pdb,configmap -o yaml \
>   > /tmp/cutover/linode-pre-cutover-aux.yaml
> ```

```bash
# On the Linode/LKE side — switch ingress to maintenance page
kubectl --context=linode-prod -n default apply -f hetzner/maintenance-ingress.yaml
# (or, in Cloudflare, deploy a Workers route returning the splash for admin/app hosts —
#  preferred because it doesn't require Linode to be healthy)

# Scale all Linode app deployments to 0 — stops further DB writes
kubectl --context=linode-prod -n default scale deploy \
  auth-service admin-core-service media-service assessment-service \
  community-service notification-service ai-service --replicas=0
kubectl --context=linode-prod -n default wait --for=delete pod \
  -l 'app in (auth-service,admin-core-service,...)' --timeout=120s

# Sanity: confirm no app pod is still running
kubectl --context=linode-prod -n default get pods | grep -E 'service'
```

**Gate:** maintenance page live + 0 app pods running → proceed.

### T+5 — Final dump from Linode

Dump in **`-Fc` custom format** (parallel-restore-capable) and run **all DBs in parallel**
so total wall-clock ≈ time-of-biggest-DB. Expected total: ~5 min for ~10 GB.

**Use the harness — `dump-restore.sh` — not ad-hoc commands.** It writes to
`~/vacademy-cutover/<ts>/` (survives laptop reboot, unlike `/tmp/cutover` which is tmpfs
on many laptops), enforces version checks, parallelism guards, and idempotent resume.

```bash
# From a jumpbox with network access to BOTH Linode and Hetzner:
cd vacademy_devops/hetzner

# Make sure standalone is OFF on Hetzner before restore (it should already be):
ssh root@<k3s-node-1> 'kubectl scale deploy --all -n default --replicas=0'

# Run the dump pipeline (preflight checks Postgres versions + connectivity,
# dump runs all 6 DBs in parallel, safety_snapshot writes a pg_dumpall .sql.gz,
# scp_to_primary ships the dumps to <db-primary>:/tmp/):
./dump-restore.sh preflight
./dump-restore.sh dump
./dump-restore.sh safety_snapshot
./dump-restore.sh scp_to_primary
```

The `linode-pre-cutover.sql.gz` safety snapshot lands under `~/vacademy-cutover/<ts>/`.
**Keep this file for at least 30 days.**

> **Fall-back appendix (raw commands)** — use ONLY if `dump-restore.sh` is broken and you
> have no time to fix it. These do NOT survive a laptop reboot and lack the harness's
> safety checks; prefer the script above.
>
> ```bash
> mkdir -p /tmp/cutover && cd /tmp/cutover
> for db in auth_service admin_core_service assessment_service media_service \
>           notification_service community_service; do
>   PGPASSWORD="$LINODE_DB_PASSWORD" pg_dump \
>     -h "$LINODE_DB_HOST" -U linroot -d "$db" \
>     -Fc --no-owner --no-privileges \
>     -f "/tmp/cutover/$db.dump" \
>     --verbose 2> "/tmp/cutover/$db.log" &
> done
> wait
> ls -lh /tmp/cutover/*.dump
> PGPASSWORD="$LINODE_DB_PASSWORD" pg_dumpall \
>   -h "$LINODE_DB_HOST" -U linroot \
>   -f /tmp/cutover/linode-pre-cutover.sql 2>/dev/null
> gzip /tmp/cutover/linode-pre-cutover.sql
> ```

### T+30 — Data verification

Before bringing services up, confirm critical tables look right.

```bash
# On <db-primary>:
for db in auth_service admin_core_service assessment_service; do
  echo "=== $db ==="
  sudo -u postgres psql -d "$db" -c "
    SELECT 'users' as t, count(*) FROM users
    UNION ALL SELECT 'roles', count(*) FROM roles
    UNION ALL SELECT 'client_secret_key', count(*) FROM client_secret_key;
  " 2>/dev/null
done

# Specifically for admin_core_service:
sudo -u postgres psql -d admin_core_service -c "
  SELECT 'course', count(*) FROM course
  UNION ALL SELECT 'level', count(*) FROM level
  UNION ALL SELECT 'session', count(*) FROM session
  UNION ALL SELECT 'package_session', count(*) FROM package_session
  UNION ALL SELECT 'groups', count(*) FROM groups;
"

# Compare these counts against the Saturday-baseline numbers you should have
# captured: scripts/snapshot-row-counts.sh > /tmp/cutover/expected-counts.txt
diff <(...) /tmp/cutover/expected-counts.txt
```

**Gate:** row counts match expected (within tolerance for the last few minutes of Linode
writes) → proceed. Major mismatch → **STOP** and re-dump; investigate before flipping.

### T+40 — Helm upgrade Hetzner chart, wait for Ready

Run `helm` **from the operator's laptop** against the Hetzner kubeconfig — the chart lives
in this repo at `vacademy_devops/vacademy-services/`, not on the cluster nodes. This avoids
having to keep a chart copy in sync on `/root/vacademy-services` on node-1.

```bash
# From the operator's laptop, in the repo root:
export KUBECONFIG=~/.kube/vacademy-prod.yaml   # must contain `hetzner-prod` context
kubectl config use-context hetzner-prod

helm upgrade --install vac vacademy_devops/vacademy-services/ \
  -f vacademy_devops/vacademy-services/values.yaml \
  -f vacademy_devops/vacademy-services/values-prod-hetzner.yaml \
  -f vacademy_devops/vacademy-services/values.secret.yaml \
  --wait --timeout 15m

# Force rollout so any stale ConfigMap-only changes apply (defensive):
for d in auth-service admin-core-service media-service assessment-service \
         community-service notification-service ai-service; do
  kubectl --context=hetzner-prod rollout restart deploy "$d"
done
for d in auth-service admin-core-service media-service assessment-service \
         community-service notification-service ai-service; do
  kubectl --context=hetzner-prod rollout status deploy "$d" --timeout=300s
done

kubectl --context=hetzner-prod get pods   # all Ready
```

> **Alternative (run helm on the node):** if you must run from `k3s-node-1`, first rsync
> the chart there (e.g. via a `hetzner/deploy-chart.sh` helper):
> ```bash
> rsync -av --delete vacademy_devops/vacademy-services/ \
>   root@<k3s-node-1>:/root/vacademy-services/
> scp vacademy_devops/vacademy-services/values.secret.yaml \
>     root@<k3s-node-1>:/root/vacademy-services/values.secret.yaml
> ssh root@<k3s-node-1> \
>   'export KUBECONFIG=/etc/rancher/k3s/k3s.yaml; cd /root/vacademy-services && \
>    helm upgrade --install vac . -f values.yaml -f values-prod-hetzner.yaml \
>      -f values.secret.yaml --wait --timeout 15m'
> ```

**Gate:** all expected deployments Ready → proceed. CrashLoopBackOff → **STOP**, debug,
or call rollback (`cloudflare-dns-flip.sh --back-to-linode`).

### T+65 — DNS flip via Cloudflare


### T+90 — Window cap

If anything is still broken at T+90:

```bash
# 1. Flip DNS back to Linode
bash cloudflare-dns-flip.sh \
  --records admin.vacademy.io,app.vacademy.io,backend.vacademy.io \
  --back-to-linode --ttl 60

# 2. Restore Linode Deployments + Ingress from the T-0 snapshot.
#    Do NOT just `scale --all --replicas=2` — original replica counts vary per service
#    (auth=2, admin-core=3, etc.) AND the maintenance-ingress.yaml apply at T+0 may have
#    replaced the real ingress. Replay the YAML captured at T-0:
bash hetzner/rollback-to-linode.sh \
  --deploys  /tmp/cutover/linode-pre-cutover-deploys.yaml \
  --ingress  /tmp/cutover/linode-pre-cutover-ingress.yaml \
  --aux      /tmp/cutover/linode-pre-cutover-aux.yaml

# Equivalent manual fall-back if rollback-to-linode.sh is unavailable:
#   kubectl --context=linode-prod -n default apply -f /tmp/cutover/linode-pre-cutover-deploys.yaml
#   kubectl --context=linode-prod -n default apply -f /tmp/cutover/linode-pre-cutover-ingress.yaml
#   kubectl --context=linode-prod -n default apply -f /tmp/cutover/linode-pre-cutover-aux.yaml
```

Then post-mortem the failure. Hetzner stays running so we can debug without pressure.

## Post-cutover

| When | What |
|---|---|
| **+30 min** | Confirm pgBackRest backed up the new prod DB cleanly. `pgbackrest info` should show a current backup. |
| **+2 h** | Compare row counts on Hetzner vs the `linode-pre-cutover.sql.gz` snapshot — sanity that nothing was lost in dump/restore. |
| **+24 h** | DB metrics check: connection count, slow queries, replication lag. |
| **+72 h** | If everything's healthy: decommission Linode K8s cluster + managed DB. **Keep `linode-pre-cutover.sql.gz` for 30 days minimum.** |
| **+1 week** | Restore Hetzner from a pgBackRest backup to a scratch box, prove the backup chain works. |

## Rollback decision tree

| Symptom | Decide by | Action |
|---|---|---|
| Helm upgrade fails / pods CrashLoop | T+55 | Rollback DNS flip not needed — we haven't flipped yet. Investigate, retry, or call window. |
| Data verification fails | T+35 | **STOP**. Re-dump from Linode (already drained), investigate before flipping. |
| Internal smoke test fails | T+65 | DNS not flipped yet — keep Linode serving via maintenance page (it's still down). Debug or rollback by bringing Linode app pods back up. |
| External smoke test fails | T+75 | Flip DNS back via `cloudflare-dns-flip.sh --back-to-linode`. Linode pods are still scaled to 0 — scale them back up. Data drift = whatever brief window of Hetzner writes happened (rare). |
| Subtle issue surfaces post-window | T+90+ | If fixable in <30 min on Hetzner: fix forward. Otherwise rollback DNS, scale Linode, debug from a position of stability. |
